# Server PRレビュー Issue草案

作成日: 2026-03-15
対象ブランチ: nasu/server
比較対象: main

## Issue 1: 切断プレイヤーのベイがゲーム状態に残留する

- 優先度: Critical
- 概要:
  - プレイヤー切断時に入力状態とジェスチャー状態は削除されるが、`game.beys[socket.id]` が削除されない。
  - 残留ベイが物理更新・衝突判定・勝敗判定に参加し続ける可能性がある。
- 影響:
  - ゲーム終了条件が満たされない、もしくは不正な勝敗になる。
  - メモリリーク/状態不整合の温床になる。
- 対象:
  - `server/src/socket/socketHandler.ts`
  - `server/src/game/gameTick.ts`
- 対応案:
  - 切断時に所属ルームのゲームを取得し、`game.beys[socket.id]` を削除する。
  - あわせて、プレイヤー離脱時の勝敗再判定を行う。
- 受け入れ条件:
  - プレイヤー切断直後に `game.beys` から当該プレイヤーが消える。
  - 2人対戦で片方が切断した場合、ゲームが正しく終了/勝者確定できる。

## Issue 2: ゲーム開始後に参加したプレイヤーにベイが生成されない

- 優先度: High
- 概要:
  - ベイ生成は `START_GAME` 時点の参加者のみ対象。
  - 開始後 `JOIN_ROOM` した参加者はルームには入るが、`game.beys` に存在しない。
- 影響:
  - 参加できたように見えて、操作/発射が機能しない。
  - UX劣化、問い合わせ増加。
- 対象:
  - `server/src/socket/socketHandler.ts`
- 対応案:
  - 仕様を決める: 開始後参加を拒否する or 途中参加時にベイを生成する。
  - 拒否する場合は `JOIN_ROOM` で `game.status === 'playing'` を検知しエラー返却。
- 受け入れ条件:
  - 開始後参加の挙動が仕様通りに固定される。
  - クライアントに明示的な成功/失敗イベントが返る。

---

## Issue 3: launchBeyが無効状態でサイレント失敗する

- 優先度: High
- 概要:
  - `LAUNCH_BEY` で `game` または `bey` が存在しないケースでもエラーを返さず終了する。
- 影響:
  - クライアント側では「発射したのに反応しない」状態になる。
  - デバッグが難しくなる。
- 対象:
  - `server/src/socket/socketHandler.ts`
- 対応案:
  - `game` 不在、`bey` 不在、未参加などの分岐で `ServerEvents.ERROR` を返す。
  - ログに roomId/playerId と失敗理由を残す。
- 受け入れ条件:
  - 無効な発射要求に対して必ず失敗理由が返る。
  - サーバーログに原因が追跡可能な形で残る。

---

## Issue 4: roomManagerのprivate内部状態へ直接アクセスしている

- 優先度: Medium
- 概要:
  - `Object.keys(roomManager['rooms'])` で内部実装に依存している。
- 影響:
  - カプセル化破壊により、将来のリファクタで破綻しやすい。
- 対象:
  - `server/src/game/gameTick.ts`
  - `server/src/room/roomManager.ts`
- 対応案:
  - `roomManager.getActiveRoomIds()` のような公開メソッドを追加する。
- 受け入れ条件:
  - `gameTick` から private フィールドへの直接アクセスが消える。
  - ルーム列挙API経由で同等動作する。

---

## Issue 5: CORSが全オリジン許可のまま固定されている

- 優先度: Medium
- 概要:
  - `origin: '*'` 固定で、公開環境でも任意オリジンから接続可能。
- 影響:
  - 本番運用時のセキュリティリスク増加。
- 対象:
  - `server/src/index.ts`
- 対応案:
  - `ALLOWED_ORIGINS` 環境変数を導入し、開発/本番で切り替える。
  - 開発時のみワイルドカードを許可する。
- 受け入れ条件:
  - 本番環境で許可オリジンが制限される。
  - 許可外オリジンは接続拒否される。

---

## 補足（テスト観点）

以下は別Issue化を推奨:

- 切断時クリーンアップの統合テスト不足
- 開始後参加シナリオのテスト不足
- 無効payload/不正型入力のランタイムバリデーションテスト不足

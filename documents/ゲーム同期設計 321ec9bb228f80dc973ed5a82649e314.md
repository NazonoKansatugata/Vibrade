# ゲーム同期設計

このゲームでは **「誰がゲーム状態を決めるか」**と **「どのデータを送るか」**を決めるのがゲーム同期設計の核心です。

スマホコントローラ型ゲームの場合、安定させるために **サーバ権威型 + 入力同期方式**が最も安全です。

---

# 1. 同期方式の方針

## サーバ権威型（Server Authoritative）

**サーバが唯一の正しいゲーム状態を持つ。**

```
スマホ → 入力送信
            ↓
         Server
      (物理計算・判定)
            ↓
        状態配信
            ↓
      PCゲーム画面
```

メリット

- 衝突ズレが起きない
- チート防止
- 同期が安定

今回の構成では **Node.jsサーバがゲームエンジン役**になります。

---

# 2. 同期の種類

ゲーム同期は主に3種類あります。

| 方式 | 内容 | 今回 |
| --- | --- | --- |
| 状態同期 | 位置を全部送る | △ |
| 入力同期 | 操作だけ送る | ◎ |
| ロック同期 | 全員同時tick | × |

今回最適なのは

**入力同期 + 状態ブロードキャスト**

# 4. Tick設計

ゲームは **Tick（更新周期）**で動きます。

例

```
60fps = 16ms
30fps = 33ms
```

今回おすすめ

```
server tick
30fps
```

理由

- ネットワーク負荷軽い
- スマホセンサー十分

---

# 5. Tickループ

サーバ側

```
Game Loop

1 入力受信
2 物理計算
3 衝突判定
4 ゲーム状態更新
5 クライアントへ配信
```

Node例

```jsx
setInterval(gameLoop, 33)
```

# 6. 入力同期設計

スマホ → サーバ

送るのは **操作だけ**

例

```json
{
  "playerId": "p1",
  "tiltX": 0.32,
  "tiltY": -0.41,
  "timestamp": 123456
}
```

頻度

```
20〜30Hz
```

# 7. 状態同期

```json
      "id": "b1",

      "y": 240,
      "vx": 2.3,
      "vy": -1.2
    }
  ]
}
```

PCは

**描画だけ**

行います。
